# 持续偏好学习闭环设计

日期：2026-06-21

## 1. 最简版本（MVP）

为 YouTube 与 AI HOT 卡片统一增加低摩擦的 👍/👎 反馈，原始反馈立即保存；每日只处理新增反馈并轻量调整短期偏好，每周把重复出现的信号固化为长期偏好，再供第二天的内容筛选使用。

## 2. 目标与非目标

### 目标

- AI HOT 和 YouTube 使用同一套反馈数据结构与学习逻辑。
- 用户每条内容只需点击 👍 或 👎，不需要选择原因或输入文字。
- 每日更新能在下一次推送中体现近期兴趣变化。
- 每周更新能识别稳定偏好、新兴兴趣和持续反感，并抑制单次误点导致的过拟合。
- Agent、Loop Engineering、Agentic Engineering 及硅谷 AI 工程/产品趋势保留独立的前沿探索通道，避免推荐越学越窄。
- 所有偏好更新可追溯、幂等、可回滚。

### 非目标

- 不增加独立数据库、向量库或新后端服务。
- 不要求用户填写点踩原因或自由文本。
- 不让自动化直接修改全局 `~/.agent/USER.md`；自动学习只更新项目的数据分支状态。
- MVP 不建设偏好管理后台，也不额外发送周报卡片。

## 3. 方案选择

### 采用：每日轻更新 + 每周深复盘

- 点击反馈立即写入原始事件。
- 每日运行只消费未处理事件，更新短期权重并生成排序提示。
- 每周运行汇总最近 7 天证据，把满足稳定条件的信号写入长期偏好。
- 一次点击只形成弱信号；至少来自 2 条不同内容、净方向达到 2，才升级为稳定偏好。

### 未采用：仅每日更新

响应快，但单次误点会被放大，容易把局部内容偏好误学成长期主题偏好。

### 未采用：仅每周更新

稳定但纠偏过慢，今天点踩的低质量内容可能连续出现数天。

## 4. 架构

```text
YouTube / AI HOT 候选
        │
        ▼
内容选择器（长期画像 + 短期偏好 + 前沿趋势通道）
        │
        ▼
飞书卡片：原文/观看 + 👍 + 👎
        │
        ▼
现有反馈 Worker
        │ 写入
        ▼
GitHub data 分支
  feedback.json            原始反馈事件
  preference_state.json    已处理游标、短期/长期权重、周快照
  profile.json             稳定用户画像与兼容字段
  ranking_hints.txt        供内容选择器直接消费的人话提示
        │
        ├─ 每日：增量分类 → 短期权重 → ranking_hints
        └─ 每周：7 日聚合 → 稳定偏好 → 周快照 → ranking_hints
```

## 5. 核心组件

### 5.1 通用反馈元数据

卡片动作统一提交以下小体积字段，避免把长摘要重复塞进飞书按钮：

- `content_id`：YouTube 视频 ID 或 `aihot:<item_id>`。
- `content_type`：`youtube` 或 `aihot`。
- `title`：内容标题。
- `creator`：频道名或 AI HOT 来源名。
- `url`：原文/视频链接。
- `category`：AI HOT 类别；视频可为空。
- `selection_tags`：选择器当时识别出的主题标签，最多 5 个。
- `action`：`like` 或 `dislike`。

Worker 同时兼容现有 `video_id`、`author`、`video_meta` 字段，历史反馈无需迁移。

### 5.2 原始反馈存储

`feedback.json` 继续按内容 ID 聚合，每次点击追加一条 reaction：

- `event_id`：Worker 生成的 UUID；历史数据缺失时由分析器根据内容 ID、时间戳和动作生成稳定哈希。
- `reaction`：`like` 或 `dislike`。
- `timestamp`：UTC ISO 时间。

原始反馈只追加，不在学习过程中覆盖或删除。

### 5.3 偏好状态

新增 `preference_state.json`：

- `schema_version`：固定为 1。
- `processed_events`：已处理事件哈希及时间，用于幂等。
- `short_term`：近 7 天主题、内容形式、来源、价值类型的权重与证据数。
- `long_term`：周度确认后的稳定偏好。
- `last_daily_run`、`last_weekly_run`：运行时间。
- `weekly_snapshots`：最近 8 份周快照，支持审计和回滚。

### 5.4 每日轻更新

每日任务在推送前执行：

1. 找出未出现在 `processed_events` 中的反馈。
2. 使用 Gemini 批量把内容映射到以下维度：
   - 主题：Loop Engineering、Coding Agent、GEO、广告创意、AI 产品等，允许产生新主题。
   - 形式：实战教程、行业新闻、产品发布、案例、访谈、工作流。
   - 价值：前沿趋势、实用方法、产品判断、商业启发、浅层宣传。
3. 👍 为相关维度增加 1，👎 减少 1；不把单条反馈扩大为整个父级主题结论。
4. 短期权重按自然日乘以 0.9 衰减。
5. 写入 `preference_state.json`，并生成新的 `ranking_hints.txt`。

Gemini 不可用或返回非法结构时，回退到确定性关键词分类；无法分类的反馈保留为未处理，下次重试，不伪造学习结果。

### 5.5 每周深复盘

每日脚本检测到距离 `last_weekly_run` 已满 7 天时自动执行周度复盘，不新增第二个定时工作流：

1. 汇总最近 7 天不同内容上的反馈证据。
2. 同一维度至少覆盖 2 条不同内容，且净方向绝对值达到 2，才更新长期偏好。
3. 长期权重每周乘以 0.95，防止旧偏好永久锁死推荐。
4. 单个来源的连续反馈优先影响来源权重，不直接否定整个主题。
5. 写入一份周快照，并重建 `ranking_hints.txt`。

### 5.6 内容选择器

YouTube 与 AI HOT 都消费同一份 `ranking_hints.txt`，但保留各自选择逻辑。

AI HOT 采用三条独立入选通道，满足任一条即可进入质量评审：

1. **Agent 优先**：Agentic/Loop/Harness Engineering、Coding Agent、MCP、Skills、Sub-agent、评测、记忆和工作流；实战教程允许入选，普通安装/API/逐行代码教程不入选。
2. **硅谷趋势**：AI 工程与产品圈正在快速传播的新概念、新工具和新工作方式。
3. **业务价值**：GEO、AI 产品、广告创意、海外增长和商业化。

最终由 Gemini 根据相关性、信息增量、实用价值、信源质量做保留/淘汰判断。最多 7 条，不设最低数量。模型失败时使用保守规则，只保留高质量且命中强主题的内容，不回退为“有什么推什么”。

为防止反馈闭环过度收窄，AI HOT 每天最多保留 2 个前沿探索位；这些内容仍需通过质量门槛，但不要求命中现有长期偏好。

### 5.7 卡片反馈体验

- YouTube 保持“观看视频 / 👍 / 👎”。
- AI HOT 改为“查看原文 / 👍 / 👎”。
- 点击后 Worker 返回飞书 toast，明确显示“已记录：有用/不想看”。
- MVP 不重绘整张混合卡片，避免现有 Worker 的视频卡片重建逻辑误删 AI HOT 内容。

## 6. 数据流

1. 内容选择器为每条入选内容生成主题标签和选择原因。
2. 飞书卡片把小体积元数据放入按钮 action value。
3. Worker 验证动作后追加写入 `feedback.json`。
4. 下一次每日任务恢复 data 分支文件，运行增量偏好分析。
5. 新偏好生成 `ranking_hints.txt`，随后参与当次 YouTube 和 AI HOT 选择。
6. 任务结束时保存 `feedback.json`、`preference_state.json`、`profile.json`、`ranking_hints.txt` 回 data 分支。

## 7. 技术栈

- 飞书交互卡片：沿用当前 Card JSON 与回调协议。
- 回调服务：沿用 Cloudflare Worker / Vercel Handler 双实现，不新增服务。
- 原始数据与状态：沿用 GitHub `data` 分支 JSON 文件。
- 分类与周度归纳：沿用现有 `google-genai` 与 `GEMINI_API_KEY`。
- 确定性回退：Python 关键词/规则分类。
- 调度：沿用 `.github/workflows/digest.yml`，通过状态时间戳触发周度复盘。

## 8. 搭建顺序

1. 为偏好状态、增量事件和周度阈值补回归测试。
2. 把 `update_preferences.py` 改为幂等增量分析，并加入每日/周度模式。
3. 扩展 Worker 的通用内容元数据和事件 ID，同时保持旧视频字段兼容。
4. 为 AI HOT 卡片增加 👍/👎。
5. 让 AI HOT 质量选择器消费偏好状态和三条入选通道。
6. 更新 workflow 恢复/保存新状态文件，并向分析步骤注入 Gemini 配置。
7. 本地测试、Worker 代表性回调测试、GitHub Actions 手动运行、飞书真实点击验证。

## 9. 边界情况与处理

- **同一按钮重复点击**：每次点击保留原始事件，但每日学习只按唯一 `event_id` 处理一次；同一内容只取最近 3 次反应，避免刷权重。
- **Actions 一天运行两次**：`processed_events` 保证旧反馈不被重复累计。
- **Gemini 不可用**：使用确定性分类；无法可靠分类的事件留待下次，不改变偏好。
- **一次误点**：单次反馈只影响短期小权重，不进入长期偏好。
- **主题与形式冲突**：例如喜欢 Agent、但不喜欢入门教程，只降低“入门教程”形式，不降低 Agent 父主题。
- **反馈文件并发写冲突**：沿用 Worker 的 409 重读重试机制。

## 10. 测试与验收

### 自动测试

- AI HOT 卡片包含“查看原文 / 👍 / 👎”，YouTube 卡片保持兼容。
- 新旧反馈 schema 均可解析。
- 同一反馈重复运行两次，第二次状态不变化。
- 一次点赞不会直接生成长期偏好。
- 两条不同内容的同向证据达到阈值后，周度复盘更新长期偏好。
- “喜欢 Agent、点踩普通教程”不会降低 Agent 父主题。
- Loop Engineering、Deep Agents 实战保留；普通 API 教程、节日营销和无关行业转售淘汰。
- Gemini 失败时不把全部候选直接推送。

### 真实验收

- 手动运行 GitHub Actions 后，飞书 AI HOT 条目出现两个反馈按钮。
- 点击任一按钮显示成功 toast，`data` 分支 `feedback.json` 出现对应 `content_type=aihot` 事件。
- 再次运行工作流不重复累计该事件。
- `ranking_hints.txt` 能看到新增短期偏好，7 天后可生成周快照。

## 11. 回滚与兼容

- 旧 `feedback.json` 无需迁移，分析器使用 `content_meta`，缺失时回退 `video_meta`。
- 删除 `preference_state.json` 可重建学习状态，但不会删除原始反馈。
- 如新选择器效果不佳，可回滚代码；旧 profile 与 ranking hints 仍可工作。
- Worker 部署失败时，卡片原文/视频链接仍可使用；反馈按钮仅显示失败，不影响主推送。

## 12. 扩展策略与潜在瓶颈

当前数据量很小，GitHub JSON 足够。增长 10 倍后，首个瓶颈会是 Worker 每次读写整个 `feedback.json` 以及 `processed_events` 持续增长。届时可把反馈迁移到 Cloudflare D1，并按事件表增量查询；在此之前不引入数据库。

最脆弱的假设是 Gemini 能稳定输出结构化主题分类。设计通过确定性回退、未分类事件重试和长期偏好阈值降低该风险。
